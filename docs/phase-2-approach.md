# Phase 2: End-to-End Data Pipeline — Approach & Execution Plan

> Phase 1 built a static demo: 9 API endpoints, SQLite for operational data, BigQuery for analytical queries, deployed on Cloud Run. Phase 2 transforms it into a live, self-sustaining data pipeline — real traffic, real data collection, automated ETL, and funnel analytics — all on GCP free tier.

---

## The Problem Phase 2 Solves

Phase 1 has two gaps that limit its portfolio value:

1. **No real data flow.** The 500K BigQuery rows were generated by a script and uploaded via `bq load`. There's no actual pipeline moving data from operational to analytical storage.
2. **No live traffic.** The API sits idle on Cloud Run (which scales to zero). There's nothing generating requests, nothing capturing interactions, nothing to analyze.

Phase 2 closes both gaps by building the infrastructure that makes data flow continuously through the system.

## Architecture: Before & After

```
PHASE 1 (Static):                    PHASE 2 (Live Pipeline):

[Cloud Run]                           [Cloud Shell]
  └── FastAPI + SQLite                  └── Locust Traffic Simulator
                                               │ (real internet traffic)
[generate_data.py]                             ▼
       │                              [e2-micro VM]
  [manual bq load]                      ├── API Container (FastAPI + Middleware)
       │                                ├── ETL Container (APScheduler)
       ▼                                └── SQLite (live request logs)
  [BigQuery]                                   │
    └── 500K static rows               [Automated ETL every 6 hours]
                                               │
                                               ▼
                                        [BigQuery]
                                          ├── recruiter_queries (500K static)
                                          └── pipeline_queries (growing live data)
```

**Key design choice:** Traffic originates from Cloud Shell (a separate machine), not the VM. Requests travel over the real internet — the middleware captures real external IPs, actual network latency, and genuine HTTP behavior. This is how production traffic works.

## The Recruiter Funnel Model

The data pipeline doesn't just capture generic web traffic — it models a **recruiter interaction funnel** that mirrors campaign measurement patterns:

| Funnel Stage | What Happens | Endpoint(s) | % of Sessions |
|-------------|-------------|-------------|---------------|
| **Impression** | Recruiter views the resume | `GET /resume` | 100% |
| **Engagement** | Recruiter explores specific sections | `GET /resume/experience`, `/skills`, `/education`, `/certifications` | ~55% |
| **Contact** | Recruiter takes action (shortlist or interview request) | `POST /resume/shortlist`, `POST /resume/contact` | ~15% |

Every request carries custom headers that the middleware captures:

| Header | Example | What It Tracks |
|--------|---------|----------------|
| `X-Session-ID` | `sess_a4f8c2e1` | Groups requests into a recruiter session |
| `X-Search-Campaign` | `senior-cloud-eng-dc` | Which recruiting search led them here |
| `X-Traffic-Source` | `linkedin` | Where they found the resume |
| `X-Funnel-Stage` | `impression` / `engagement` / `contact` | Stage in the interaction journey |
| `X-Device-Type` | `desktop` | Device segmentation |
| `X-Geo-Region` | `us-east` | Geographic segmentation |

This creates a dataset with session-level attribution, campaign performance metrics, channel conversion rates, and funnel drop-off analysis — the same analytical patterns used in digital advertising measurement.

## Technology Decisions

| Decision | Choice | Why | Alternative Rejected |
|----------|--------|-----|---------------------|
| Hosting | e2-micro VM | Always Free, persistent 24/7, runs scheduled jobs | Cloud Run (ephemeral, scales to zero — can't run ETL) |
| Container orchestration | Docker Compose | 2 services (API + ETL), simple, single-host | Docker run (no health checks or dependency management) |
| Traffic simulation | Locust from Cloud Shell | Real internet traffic, weighted funnel behavior, free | Locust on same VM (localhost traffic — not realistic) |
| ETL scheduling | APScheduler | Unlimited free jobs, Python-native, runs in container | Cloud Scheduler (only 3 free jobs) |
| BigQuery loading | Batch (`load_table_from_dataframe`) | Free tier compatible | Streaming inserts ($0.005/100K rows — not free) |
| Request logging | `BackgroundTask` on response | Non-blocking, <50ms latency impact | Synchronous writes (blocks response) |
| Sync tracking | `synced_to_bq` column | Prevents duplicate loads, exact tracking | Timestamp-based (fragile, gaps possible) |
| Security | fail2ban + non-root containers | Minimal attack surface, standard practice | Complex iptables rules (harder to maintain) |

## What the Pipeline Produces

### 10 BigQuery Queries (5 Pipeline + 5 Funnel)

**Pipeline Health Queries:**
1. Top recruiter domains (last 24 hours)
2. Hourly traffic distribution
3. Most viewed endpoints
4. Pipeline health check (records per day)
5. Skill search analysis

**Funnel Analysis Queries:**
6. Funnel conversion rate by campaign (CTE + SAFE_DIVIDE + COUNTIF)
7. Funnel drop-off analysis (where recruiters abandon)
8. Conversion rate by traffic source (channel attribution)
9. Campaign performance by device & region (cross-segmentation)
10. Session engagement depth (do engaged recruiters convert more?)

### Data Schema: `pipeline_queries` Table (BigQuery)

| Column | Type | Source |
|--------|------|--------|
| `query_id` | INTEGER | Auto-increment |
| `request_timestamp` | TIMESTAMP | Middleware |
| `http_method` | STRING | GET or POST |
| `endpoint_path` | STRING | /resume, /resume/skills, etc. |
| `query_params` | STRING | ?keyword=python, etc. |
| `recruiter_domain` | STRING | Extracted from Referer header |
| `user_agent` | STRING | Browser/client identifier |
| `client_ip` | STRING | Real external IP (from Cloud Shell) |
| `http_status_code` | INTEGER | 200, 404, etc. |
| `response_time_ms` | FLOAT | End-to-end latency |
| `session_id` | STRING | X-Session-ID header |
| `search_campaign` | STRING | X-Search-Campaign header |
| `traffic_source` | STRING | X-Traffic-Source header |
| `funnel_stage` | STRING | impression / engagement / contact |
| `device_type` | STRING | desktop / mobile / tablet |
| `geo_region` | STRING | us-east, us-west, etc. |
| `hour_of_day` | INTEGER | Derived: 0-23 |
| `day_of_week` | STRING | Derived: Monday-Sunday |
| `date` | DATE | Derived: for partitioning |

## End State

When complete, the system runs autonomously:

```
[Cloud Shell: Locust] ──internet──> [VM: FastAPI + Middleware] → [SQLite] → [APScheduler ETL] → [BigQuery] → [Analytics]
```

- Traffic flows over the real internet
- Every request is captured with funnel context
- Data syncs to BigQuery every 6 hours
- 10 analytical queries reveal traffic patterns, funnel performance, and campaign attribution
- VM auto-starts on reboot (systemd)
- SSH is protected (fail2ban)

This is the same operational-to-analytical pipeline pattern used in ad tech: event capture, operational store, batch ETL, analytical warehouse, reporting. The difference is scale — but the architecture is identical.
